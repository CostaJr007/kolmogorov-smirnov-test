<methodology_and_domain_expertise>

  <methodology>
  **Gray-box Injection Analysis (Information Theory Enhanced)**

  - **Goal:** Identify injection points by detecting anomalies in server responses using Shannon Entropy and Information Gain.
  - **1) Create a To Do for each Injection Source found in the Pre-Recon Deliverable**
  - **2) Establish Baseline:**
      - Send a valid, benign request to the target endpoint.
      - Capture the response body, status code, and response time.
      - This is your `baseline_body`.
  - **3) Fuzzing & Differential Analysis:**
      - Send requests with injection payloads (SQLi, CMDi, etc.).
      - Capture the `target_body`.
      - Use `analyze_response(baseline_body, target_body)` to get anomaly metrics.
  - **4) Interpret Results:**
      - **High Information Gain (> 0.8):** The response structure has changed significantly. This could be a WAF block page, an error page, or a successful injection that altered the query result.
      - **High Entropy Diff (> 1.0):** The randomness of the response changed. A sudden drop might indicate a simple error page replacing complex content. A rise might indicate a data dump.
      - **Time Delay:** If the response time increases significantly (e.g., > 5s for a `SLEEP(5)` payload), it's a strong indicator of blind injection.
  - **5) Make the call (vulnerability or safe)**
      - **Vulnerable:** Significant anomaly detected that correlates with the payload type (e.g., time delay for time-based payload, syntax error for syntax payload).
      - **Safe:** No significant deviation from baseline, or deviation is clearly a standard validation error (400 Bad Request) without sensitive info.
  - **6) Append to findings list (consistent fields)**
      - Use the Write tool to log the finding in `deliverables/injection_exploitation_queue.json`.
  </methodology>

</methodology_and_domain_expertise>
